# 后端代码审查分工（5 人）

本文档用于将后端业务逻辑审查拆分为 5 个可并行、低重叠的任务包。原则：**每人负责一条完整链路，模块交叉点单独联审**。

参考文档：

- [智能充电桩调度计费系统详细需求.md](./智能充电桩调度计费系统详细需求.md)
- [API接口文档.md](./API接口文档.md)
- [后端架构参考手册.md](./后端架构参考手册.md)
- [作业验收用例.md](./作业验收用例.md)

---

## 分工总览

| 负责人 | 模块 | 主要文件 | 审查重点 |
|--------|------|----------|----------|
| **A** | 基础设施 + 认证 | `common/*`、`accounts/*` | JWT、鉴权、枚举/错误码、分时计费工具、模拟时钟 |
| **B** | 充电站 + 桩管理 | `station/*`、`init_system` | 系统配置、桩状态机、启停桩、初始化 |
| **C** | 排队 + 调度 | `charging` 中 Queue/Dispatch | 等候区/桩队列、调度策略、选桩算法 |
| **D** | 请求 + 会话 + 计费 | `charging` 中 Request/Session、`billing/*` | 状态流转、启停充、详单/账单 |
| **E** | 运维 + 故障 + 验收 | `operations/*`、`run_acceptance` | 故障/重调度、报表、验收自动化 |

## A — 基础设施与认证

### 文件清单

```text
backend/apps/common/
  auth.py, decorators.py, enums.py, exceptions.py
  models.py, responses.py, sim_clock.py, utils.py
backend/apps/accounts/
  models.py, services.py, views.py, urls.py
backend/config/__init__.py          # PyMySQL
backend/config/settings/base.py     # JWT、CORS、INSTALLED_APPS
```

### 必查逻辑

- [ ] 注册：车牌唯一，同时创建 `UserAccount` + `Vehicle`，Token 含 `vehicle_id`
- [ ] 登录：先查 `AdminAccount`，再查 `Vehicle`；密码 bcrypt 校验
- [ ] `@require_user` / `@require_admin` 装饰器是否正确挂载到各 View
- [ ] `ErrorCode` 与 API 文档、业务抛错是否一致
- [ ] `compute_charged_amount`：min(功率×时长, 请求电量)
- [ ] `calculate_cross_period_fee`：峰/平/谷跨时段拆分是否正确
- [ ] `system_now()`：模拟时钟开启/关闭时行为是否正确
- [ ] `api_view` 统一捕获 `AppException` 与非预期异常

### 建议对照测试

`apps.tests.test_all_apis` 中 **认证** 相关用例（`test_auth_*`）

## B — 充电站与桩管理

### 文件清单

```text
backend/apps/station/
  models.py, services.py, views.py, urls.py
  management/commands/init_system.py
```

### 必查逻辑

- [ ] 桩状态机：`off → standby → available ↔ charging`，故障为 `fault`
- [ ] `power_on`：仅 `off/standby` 可上电 → `standby`
- [ ] `start_pile`：投入运行 → `available`，并触发 `DispatchService.try_dispatch_all()`
- [ ] `power_off`：无排队票、无在充会话才可关闭
- [ ] `SystemConfig` 默认值：N=10、M=3、2 快充 + 3 慢充
- [ ] `update_config` 允许字段与枚举校验（`fault_strategy`、`dispatch_mode`）
- [ ] `init_system`：管理员、桩、计费策略初始化是否完整

### 建议对照测试

`test_admin_system_config`、`test_admin_pile_status`、`test_admin_pile_lifecycle`

## C — 排队与调度

### 文件清单

```text
backend/apps/charging/models.py
  ChargingRequest, QueueTicket, DispatchRecord
backend/apps/charging/services.py
  QueueService（全文）
  DispatchService（全文）
```

> `charging/services.py` 中与 C 相关的类约第 1–286 行；D 负责同文件后半部分。

### 必查逻辑

- [ ] 排队号：快充 `F{n}`、慢充 `T{n}`，改模式后重新取号并重排
- [ ] 等候区满判定：`waiting_area_used >= waiting_area_size` 且同模式桩无空位 → `accepted: false`
- [ ] `waiting_dispatch_paused` 为 true 时，`try_dispatch_mode` 是否停止叫号
- [ ] **normal**：FIFO + 最短总完成时间选桩
- [ ] **single_min_total**：按可用槽位数批量派单
- [ ] **batch_min_total**：系统满载后全局最优分配
- [ ] `_estimate_pile_completion_time`：在充剩余 + 队列等待时间估算
- [ ] `pile_occupancy_count`：在充 + dispatched + pending_reschedule
- [ ] `_dispatch_request_to_pile`：写 DispatchRecord、更新车辆状态、创建桩队列票

### 建议对照测试

多车提交、模式切换、暂停调度（`_pause_dispatch`）相关场景；与 D 联查 `submit_request` 入口。

## D — 充电请求、会话与计费

### 文件清单

```text
backend/apps/charging/services.py
  ChargingRequestService（全文）
  ChargingSessionService（全文）
backend/apps/charging/views.py
backend/apps/charging/urls.py
backend/apps/billing/
  models.py, services.py, views.py, urls.py
```

> `charging/services.py` 中与 D 相关的类约第 288 行起至文件末尾。

### 必查逻辑

- [ ] 状态机：`queuing → dispatched → charging → completed/cancelled`
- [ ] 故障分支：`pending_reschedule` 的处理与 C/E 衔接
- [ ] 修改充电量/模式：仅 `queuing` 可操作；改模式重取号
- [ ] 开始充电：须 `dispatched`、队首、`pile_id` 一致、桩无并发会话
- [ ] `_finalize_session`：更新桩统计、调用 `BillingService.create_detail_and_bill`、再调度
- [ ] 取消：排队/派队直接取消；在充则按已充电量出账
- [ ] 等候区满：HTTP 返回 `accepted: false`（非 3005），与文档一致
- [ ] 计费：充电费跨时段 + 服务费 0.8×电量；每次会话一账单一明细
- [ ] View 层只做鉴权/参数解析，不含业务分支

### 建议对照测试

`test_user_charging_flow`（若有）、`test_user_update_amount_and_mode_in_waiting_area`、`test_user_cancel_while_queuing`、账单 list/detail/pay 相关用例

## E — 故障、报表与验收

### 文件清单

```text
backend/apps/operations/
  models.py, services.py, views.py, urls.py
  acceptance_service.py, acceptance_events.py
  management/commands/run_acceptance.py
backend/apps/tests/test_acceptance_snapshot.py
```

### 必查逻辑

- [ ] 故障上报：暂停叫号 → 在充 `_finalize_session(fault)` → 队列 `pending_reschedule`
- [ ] 故障桩队列车辆**不**转入等候区，有空位优先派到同类型其它桩
- [ ] 故障恢复：合并同类型桩未充电队列，按排队号整体重调度
- [ ] `RescheduleService.sync_waiting_dispatch_pause` 与 C 的调度暂停一致
- [ ] `ReportService.generate_stats`：日/周/月聚合与 ReportSnapshot 写入
- [ ] `AcceptanceService.stabilize_system`：满电结束 → 重调度 → 自动启充
- [ ] `submit_charging_request`：被拒后重试，不中断验收（非 HTTP 3005 问题）
- [ ] `build_snapshot` / `validate_snapshot` 与作业用例表格式一致（每时刻 3 行）
- [ ] 09:00 场景：V30 到达被拒（`accepted: false` / `request_rejected`）

### 建议对照测试

```bash
python manage.py test apps.tests.test_acceptance_snapshot --settings=config.settings.test
python manage.py run_acceptance --fast --until 09:30
```

## 交叉点联审清单（第 3 轮，5 人一起）

以下逻辑跨模块，**单人审查易漏**，建议联审 1 小时逐项勾选：

| # | 交叉点 | 涉及 | 检查内容 |
|---|--------|------|----------|
| 1 | 调度触发 | B/C/D/E | 桩 `start`、结束充电、取消、故障恢复后是否都调用 `try_dispatch_all` |
| 2 | 时间源 | A/C/D/E | 业务时间是否统一 `system_now()`，报表/验收/计费是否一致 |
| 3 | 队列计数 | C/D/E | `pile_occupancy_count` 与 `pile.current_queue_length` 更新是否一致 |
| 4 | 故障 + 等候区 | C/E | 故障期间等候区叫号暂停；故障桩队列不进等候区 |
| 5 | 在充唯一性 | D/E | 每桩同时最多一个 `ACTIVE` 会话；`try_auto_start_all` 不抢充 |
| 6 | HTTP vs Service | D + 各 View | 验收/管理命令走 Service，HTTP 走 View，行为是否等价 |
| 7 | 取消在充 | D | 响应 `status` 与 DB 状态（`cancelled` vs `completed`）是否可接受 |

---

## 建议审查流程

```text
第 1 轮（并行，各 2–4h）
  A/B/C/D/E 按上文清单审各自文件，填写「审查记录表」

第 2 轮（并行，各 1–2h）
  每人阅读相邻模块一个（如 C 读 D 的 submit_request；D 读 C 的 try_dispatch_all）

第 3 轮（串行联审，约 1h）
  5 人一起过「交叉点联审清单」

第 4 轮（测试）
  全员本地跑：
    python manage.py test apps.tests --settings=config.settings.test
  E 额外跑：
    python manage.py run_acceptance --fast --until 09:30
```

## 不建议的分法

| 分法 | 问题 |
|------|------|
| 按 models / services / views 五人分 | 同一条状态机被拆散，无人端到端负责 |
| `charging/services.py` 按行数机械切半 | 易漏 `submit_request` ↔ `_dispatch_request_to_pile` 衔接 |
| 一人只审 tests | 测试覆盖有限，难以理解业务意图 |
