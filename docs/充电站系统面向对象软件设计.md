## 软件架构

### 软件架构示意图

本系统采用Vue+Django+MySQL的前后端分离B/S架构，实现智能充电桩调度与计费管理。系统划分为用户界面层、控制器层、业务/应用层、持久化层、系统支持层和数据库层。各层职责清晰，确保系统高内聚、低耦合、易扩展。

```mermaid
graph TD
    Browser[用户 / 浏览器] -->|访问系统| Vue[用户界面层 Vue 前端<br/>页面展示 / 用户操作 / 接口调用]
    Vue -->|HTTP / JSON| Controller[控制器层<br/>urls.py + views.py + serializers.py<br/>路由分发 / 请求接收 / 参数校验 / 响应返回]
    Controller -->|调用业务服务| Business[业务 / 应用层<br/>services.py<br/>用例流程 / 业务规则 / 调度策略]
    Business -->|读写数据对象| Persistence[持久化层<br/>models.py + Django ORM<br/>数据模型 / 对象映射 / 数据同步]
    Persistence -->|ORM / SQL| MySQL[(数据库层 MySQL<br/>用户 / 车辆 / 充电请求 / 充电桩 / 账单 / 详单)]
    Support[系统支持层<br/>settings / middleware / auth / logging<br/>配置 / 认证 / 权限 / 日志 / 异常处理]
    Support -.-> Controller
    Support -.-> Business
    Support -.-> Persistence
```

用户界面层（Vue前端）

提供注册、充电申请、账单/详单查询、充电桩管理、队列监控等功能的交互界面；仅负责采集用户输入和展示系统返回信息，不处理核心业务逻辑。

控制器层（Django Controller）

接收前端请求、参数校验、调用业务层服务、封装返回结果。

业务/应用层（Service）

实现充电申请、调度策略、计费计算、故障恢复等核心业务逻辑。

持久化层（Django ORM）

封装数据库访问，实现对象与数据库表映射；提供增删改查操作，保证数据一致性；隔离数据库细节。

系统支持层（Support Layer）

提供日志管理、缓存服务、任务调度、消息队列、安全与认证、监控与告警等功能，为业务层和控制器层提供全局通用支持，确保系统可靠运行。

数据库层（MySQL）

存储用户信息、车辆信息、充电请求记录、充电桩状态、账单和详单数据等；通过Django ORM与系统进行交互。

架构合理性

职责划分明确：每层关注自身功能，降低模块依赖，提高可维护性。

便于扩展：新增调度策略或计费规则时仅需修改业务层，不影响前端和数据库。

开发效率高：Django ORM自动完成对象与表映射，减少SQL开发工作量。

支持并发：前端与REST接口采用HTTP/JSON通信，可支持多用户同时访问。

系统可靠性：系统支持层提供日志、缓存、任务调度、消息队列和监控，保障业务连续性。

运行机制：

用户通过Vue前端发起请求；

Django控制器层接收请求并进行参数校验；

控制器调用业务层服务执行核心逻辑（调度、计费、状态更新）；

业务层通过ORM持久化数据到MySQL；

系统支持层提供日志、缓存、异步任务和监控等支持；

控制器层封装结果返回前端，完成一次请求-响应流程。

### 分层结构说明

本系统在 1.1 所示架构基础上，按照“界面展示—请求控制—业务处理—对象持久化—数据库存储”的链路划分职责。前端页面不直接访问数据库，也不承担核心业务计算；所有操作均以 HTTP/JSON 请求进入后端控制器，再由控制器委托业务服务、领域对象和持久化对象完成处理。

（1）接收前端请求的对象

Django 控制器层负责接收 Vue 前端请求。充电申请相关请求由 ChargingView 接收；账单与详单查询由 ViewBillController 接收；充电桩启停和参数设置由 RunChargingPileController 接收；状态监控由 ChargingPileMonitorController、ChargingQueueMonitorController 接收；故障调度由 FaultDispatchController 接收。控制器只负责路由匹配、身份认证、权限校验、参数检查和统一响应封装，不直接编写复杂业务规则。

（2）创建对象实例的对象

对象创建主要由业务服务层和领域模型层完成。ChargingService、BillingService、BillService、FaultDispatchService、FaultRecoveryService 等服务负责协调用例流程，并在需要时创建 ChargingRequest、QueueTicket、FaultRecord、ChargingDetail、Bill、PileStateDTO、QueueVehicleDTO 等对象；与充电桩状态紧密相关的 ChargingSession 可由 ChargingPile 发起创建，服务层负责协调和持久化。

（3）处理请求具体要求的对象

业务/应用层负责执行具体业务。ChargingService 处理充电申请、修改请求、开始充电和结束充电；ChargingStatusService 计算充电状态和临时详单；BillingService、BillService 完成计费与账单汇总；ChargingPileService 处理充电桩启停、运行状态和计费参数；监控服务负责统计充电桩与队列状态；FaultDispatchService 结合优先级调度、时间顺序调度和正常调度策略完成故障迁移与恢复。

（4）数据同步和存储的对象

数据同步由 Django ORM、Model 以及必要的 DAO/Repository 对象完成，最终持久化到 MySQL。主要持久化对象包括 User、Vehicle、ChargingRequest、ChargingSession、ChargingPile、QueueTicket、WaitingArea、TariffPolicy、TimePeriod、ChargingDetail、Bill/Invoice、FaultRecord 等。对于结束充电、生成详单、故障迁移等跨对象操作，应通过事务保证充电状态、队列调整和账单数据的一致性。

综上，系统一次典型请求的处理流程为：Vue 前端发起请求，控制器层完成入口处理，业务/应用层调用领域模型和策略对象，持久化层通过 ORM 将结果同步到 MySQL，最后控制器将处理结果以 JSON 形式返回前端。

问题1：第一个接收该消息的软件对象是什么？

该消息由用户端发起，因此第一个接收该消息的软件对象为控制器层对象 ChargingView。ChargingView 接收请求后，将开始充电的业务逻辑委托给 ChargingService。

问题2：根据操作契约，哪个对象负责创建 ChargingSession 实例？

开始充电时创建的新实例是 ChargingSession，它表示一次真实的充电过程。根据创建者模式，ChargingPile 记录当前正在服务的会话，并且拥有创建会话所需的充电桩编号、功率、队首车辆等信息，因此由 ChargingPile 负责创建或发起创建 ChargingSession 更合理。ChargingService 负责协调流程，不直接承担领域对象创建职责。

问题3：哪些对象之间需要建立或消除关联关系？

需要建立 ChargingSession 与 ChargingPile 的关联，表示该会话占用某个充电桩；同时建立 ChargingSession 与 ChargingRequest 的关联，表示该会话来源于某个充电请求。车辆开始充电后，对应 QueueTicket 从充电桩等待队列中移除。

问题4：哪个对象的属性需要修改或初始化？

ChargingSession 初始化 startTime、chargeAmount、status 等属性。ChargingRequest.car_state 由 queued 修改为 charging。ChargingPile.currentSession 修改为当前新建的 ChargingSession。

不修改车辆状态；
6. 返回详单信息，至少包括：详单编号、详单生成时间、充电桩编号、充电电量、充电时长、启动时间、停止时间、充电费用、服务费用、总费用。 |

问题1：第一个接收该消息的软件对象是什么？

该消息由用户端发起，因此第一个接收消息的软件对象统一为控制器层对象 ChargingView。它接收 Query_Charging_State(car_id) 后，将具体查询与计算逻辑委托给 ChargingStatusService。

问题2：根据操作契约，该操作是否需要创建新的对象实例？

不需要。该指令本质上是查询型指令。如果车辆正在充电，系统返回的是根据当前会话实时计算得到的临时详单信息；如果车辆已完成充电，则读取已生成的 Invoice。查询本身不创建新的 ChargingSession 或 Invoice。

问题3：哪些对象之间需要建立或消除关联关系？

不需要建立或消除新的关联。该操作只读取已有的 ChargingSession 与 ChargingPile 关联，或者读取已有的 ChargingSession 与 Invoice 关联。

问题4：哪个对象的属性需要修改或初始化？

该操作不修改领域对象属性。ChargingStatusService 只负责计算临时详单字段，例如 ChargeDuration、ChargeFee、ServiceFee 和 TotalFee。

问题3：哪个对象负责生成充电详单？

问题4：生成详单后，谁负责持久化？

问题5：哪个对象负责修改充电请求的状态？

充电请求对象自身拥有状态属性，由控制器发送 setStatus("已完成") 消息。

问题6：如何从队列中移除该请求？

充电桩聚合了一个队列，队列负责管理请求列表。由控制器调用 queue.remove(req)。

问题7：如何调度下一辆车？

### 充电桩故障调度

#### 已知条件

| 指令 | 返回 |  |
| --- | --- | --- |
| 1、优先级故障调度 | Priority_Fault_Dispatch(pile_Id) | Return(dispatchPlan, failedList) |
| 2、时间顺序故障调度 | TimeOrder_Fault_Dispatch(pile_Id) | Return(dispatchPlan, failedList) |
| 3、故障恢复调度 | Fault_Recovery_Dispatch(pile_Id) | Return(0/1, recoveredPileState, scheduledList) |

#### 对象设计：优先级调度

操作契约

| 操作 | Priority_Fault_Dispatch(pile_Id) |
| --- | --- |
| 前置条件 | 管理员或监控服务已通过认证；
pile_Id 对应 ChargingPile 存在；
该充电桩状态为 FAULT 或设备心跳异常；
系统能够读取该桩的当前 ChargingSession 和等待队列；
同模式可用充电桩存在空位，或等候区可接收未迁移车辆。 |
| 后置条件 | 创建或更新 FaultRecord，记录故障桩、故障类型和故障时间；
将 ChargingPile.working_state 修改为 FAULT，并停止该桩监控任务；
若存在正在充电的 ChargingSession，则中断会话并保存已充电量；
按“正在充电车辆优先、队首车辆优先、等待时间长者优先”的规则排序受影响请求；
对可迁移车辆创建新的 QueueTicket 并加入目标 ChargingPile 队列；
无法迁移的请求回到 WaitingArea，保持原 request_time；
7. 返回调度方案和失败列表。 |

问题1：第一个接收该消息的软件对象是什么？

该消息可能由 PileMonitorScheduler 自动触发，也可能由管理员在监控界面手动确认触发。进入系统后的第一个接收对象为控制器层 FaultDispatchController，它负责校验 pile_Id、故障状态和管理员/系统权限，并将业务处理委托给 FaultDispatchService。

问题2：根据操作契约，哪个对象负责创建对象实例？

FaultDispatchService 负责创建 FaultRecord 和新的 QueueTicket 实例。FaultRecord 用于记录故障事件；QueueTicket 用于表示车辆迁移到目标充电桩后的新排队位置。PriorityDispatchStrategy 只负责排序和选择方案，不直接持久化领域对象。

问题3：哪些对象之间需要建立或消除关联关系？

建立 FaultRecord 与故障 ChargingPile 的关联；建立新 QueueTicket 与目标 ChargingPile 的队列关联；建立 QueueTicket 与 ChargingRequest 的关联。车辆迁移后，需要消除旧 QueueTicket 与故障 ChargingPile 队列的关联。若车辆正在充电，还需要保留 ChargingSession 与 ChargingRequest 的历史关联，用于后续详单和补偿计算。

问题4：哪个对象的属性需要修改或者初始化？

ChargingPile 的 working_state 修改为 FAULT，is_available 修改为 false；ChargingSession 的 status 修改为 interrupted，并记录 end_time 与 charged_amount；ChargingRequest 的 car_state 根据迁移结果修改为 queued 或 waiting，同时更新 queue_Num、queue_Position；新 QueueTicket 初始化 queue_Num、queue_Position、sourcePileId、targetPileId 等字段。

问题5：优先级规则如何确定？

优先级调度采用稳定排序：正在充电且被中断的车辆优先级最高；故障前位于队首的车辆优先于队尾车辆；同一优先级下按 request_time 升序排列。这样既能保护已经开始服务的车辆，也能尽量保持原排队公平性。

#### 对象设计：时间顺序调度

操作契约

| 操作 | TimeOrder_Fault_Dispatch(pile_Id) |
| --- | --- |
| 前置条件 | 1. 管理员或监控服务已通过认证；2. pile_Id 对应 ChargingPile 存在且处于故障状态；3. 故障桩队列中存在 queued 状态的 ChargingRequest；4. 本策略适用于无特殊优先级的普通故障迁移；5. 每个待迁移请求仍保留原 request_time。 |
| 后置条件 | 1. 读取故障桩队列中所有受影响请求；2. 按 ChargingRequest.request_time 升序排序；3. 依次查找同模式可用目标桩；4. 若目标桩有空位，则移除旧 QueueTicket，创建新 QueueTicket 并更新队列位置；5. 若暂时无空位，则将请求放回 WaitingArea，car_state 修改为 waiting；6. 返回迁移结果。 |

问题1：第一个接收该消息的软件对象是什么？

第一个接收消息的软件对象仍为 FaultDispatchController。该控制器接收 TimeOrder_Fault_Dispatch(pile_Id)，完成参数校验后调用 FaultDispatchService.timeOrderFaultDispatch()。

问题2：根据操作契约，哪个对象负责创建 QueueTicket 实例？

FaultDispatchService 根据 TimeOrderDispatchStrategy 返回的 orderedRequests 创建新的 QueueTicket。排序策略对象只提供排序结果，QueueTicket 的创建、删除和持久化由服务层统一协调，避免策略对象承担数据库职责。

问题3：哪些对象之间需要建立或消除关联关系？

需要消除旧 QueueTicket 与故障 ChargingPile 队列的关联；建立新 QueueTicket 与目标 ChargingPile 的队列关联；ChargingRequest 与 Vehicle 的关联保持不变。若车辆未能迁移到目标桩，则 ChargingRequest 与 WaitingArea 建立等待关联。

问题4：哪个对象的属性需要修改或者初始化？

新 QueueTicket 初始化 queue_Num、queue_Position、create_time；ChargingRequest 的 car_state 修改为 queued 或 waiting；queue_Num、queue_Position 根据目标桩队列重新计算；故障 ChargingPile 的队列长度需要同步减少。

问题5：时间顺序调度与优先级调度的区别是什么？

时间顺序调度不区分正在充电、队首、队尾等特殊优先级，只按原始 request_time 升序处理，适用于普通排队车辆迁移。优先级调度更适合存在正在充电车辆被中断、需要保护服务连续性的场景。

#### 对象设计：故障恢复调度

操作契约

| 操作 | Fault_Recovery_Dispatch(pile_Id) |
| --- | --- |
| 前置条件 | 管理员确认维修完成，或设备自检上报恢复；
pile_Id 对应 ChargingPile 存在；
该桩状态为 FAULT 或 REPAIRING；
物理设备通信链路正常；
充电桩已绑定有效 TariffPolicy；
6. 该桩不存在 status="charging" 的活跃 ChargingSession。 |
| 后置条件 | 调用设备适配器执行自检；
自检通过后清除故障标记，将 ChargingPile.working_state 修改为 IDLE 或 RUNNING；
关闭 FaultRecord，记录 recovery_time；
重新注册 PileMonitorScheduler；
若恢复桩队列存在空位，则调用 NormalDispatchStrategy 从 WaitingArea 中调入同模式车辆；
持久化充电桩状态、故障记录和新建 QueueTicket；
7. 自检失败时保持故障状态并返回失败原因。 |

问题1：第一个接收该消息的软件对象是什么？

第一个接收消息的软件对象为 FaultDispatchController。故障恢复仍属于“充电桩故障调度”用例，控制器负责接收恢复请求、确认维修权限，并调用 FaultRecoveryService 完成恢复流程。

问题2：根据操作契约，哪个对象负责创建对象实例？

FaultRecoveryService 通常不创建新的核心业务实体，只更新 ChargingPile 和 FaultRecord。当恢复后的充电桩出现空位并触发补位时，NormalDispatchStrategy 结合 DispatchService 创建新的 QueueTicket 实例。若系统需要审计，也可以由 FaultRecoveryService 创建 RecoveryLog。

问题3：哪些对象之间需要建立或消除关联关系？

恢复流程会关闭 FaultRecord 与 ChargingPile 的故障关联状态，但保留历史记录；重新建立 ChargingPile 与 PileMonitorScheduler 的监控注册关系；若触发补位，则建立 QueueTicket 与恢复桩的队列关联，并消除对应车辆在 WaitingArea 中的等待关联。已在故障期间迁移到其他桩的车辆不强制迁回，避免破坏已稳定的队列秩序。

问题4：哪个对象的属性需要修改或者初始化？

ChargingPile 的 fault_flag 修改为 false，working_state 修改为 IDLE 或 RUNNING，update_time 更新为当前时间；FaultRecord 的 status 修改为 recovered，recovery_time 初始化为当前时间；新建 QueueTicket 初始化 queue_Num、queue_Position、create_time；被调入车辆的 ChargingRequest.car_state 修改为 queued。

问题5：恢复失败时如何处理？

若设备自检失败、计费规则缺失或存在未结束的 ChargingSession，则保持 ChargingPile 的故障状态，不关闭 FaultRecord，也不触发正常调度。FaultDispatchController 将失败原因封装为 JSON 返回给管理员端。

#### 已知条件

| 指令 | 返回 |  |

#### 对象设计：单次调度总充电时长最短

操作契约

| 操作 | Single_MinTime_Dispatch() |
| --- | --- |
| 前置条件 | 充电区出现多个车辆空位；
等候区存在等待调度的车辆；
每辆待调度车辆仍处于 waiting 状态；
本策略按充电模式分配对应充电桩，即快充车辆只能进入快充桩队列，慢充车辆只能进入慢充桩队列；
5. 不考虑用户修改请求和充电桩故障。 |
| 后置条件 | 为每辆被调度进入充电区的车辆创建新的 QueueTicket 对象；
建立 QueueTicket 与目标 ChargingPile 的队列关联；
消除该车辆在 WaitingArea 中的等待关联；
修改对应 ChargingRequest 的属性：car_state = queued，新增或更新 queue_Num 与 queue_Position；
5. 返回本次调度车辆数量和调度方案。 |

问题1：第一个接收该消息的软件对象是什么？

该消息属于调度系统内部触发的扩展调度指令，不由普通用户直接触发。因此第一个接收消息的软件对象使用控制器层对象 DispatchController，由它接收空位事件或定时任务触发的 Single_MinTime_Dispatch()。

问题2：根据操作契约，哪个对象负责创建 QueueTicket 实例？

本操作中需要为进入充电区的车辆创建新的排队号 QueueTicket。从职责看，DispatchService 掌握最优调度方案中的车辆、目标充电桩编号和队列位置，因此由 DispatchService 调用 QueueTicket 的创建操作最合适。QueueTicket 创建后再加入目标 ChargingPile 的等待队列。

问题3：哪些对象之间需要建立或消除关联关系？

需要建立 QueueTicket 与目标 ChargingPile 的队列关联，即 ChargingPile 管理该排队号；同时车辆被调度进入充电区后，应从 WaitingArea 中移除，表示原有等待关联被消除。

问题4：哪个对象的属性需要修改或初始化？

新建的 QueueTicket 需要初始化 queue_Num 和 queue_Position。对应 ChargingRequest 的 car_state 需要由 waiting 修改为 queued，同时写入 queue_Num 和 queue_Position。

#### 对象设计：批量调度总充电时长最短

操作契约

| 操作 | Batch_MinTime_Dispatch() |
| --- | --- |
| 前置条件 | 到达充电站的车辆数量等于全部车位数量，即充电区容量与等候区容量之和；
当前批次尚未开始调度；
本策略不区分快充和慢充模式；
车辆到达先后顺序不参与调度；
所有车辆均可分配到任意类型充电桩；
6. 不考虑用户修改请求和充电桩故障。 |
| 后置条件 | 为批次中每辆进入充电区队列的车辆创建新的 QueueTicket 对象；
建立每个 QueueTicket 与目标 ChargingPile 的队列关联；
消除车辆与 WaitingArea 的等待关联；
修改对应 ChargingRequest 的属性：car_state = queued，新增或更新 queue_Num 与 queue_Position；
5. 返回批量调度车辆数量和调度方案。 |

问题1：第一个接收该消息的软件对象是什么？

批量调度由“车辆数达到总车位数”这一内部条件触发，因此由DispatchController作为第一个接收该系统事件的软件对象。

问题2：根据操作契约，哪个对象负责创建QueueTicket实例？

批量调度中，BatchDispatchService掌握当前批次车辆、所有可用充电桩以及最优调度方案，因此由BatchDispatchService负责调用QueueTicket的创建操作，为每辆被分配到充电区队列的车辆创建新的排队号。

问题3：哪些对象之间需要建立或消除关联关系？

每个新建的QueueTicket都需要加入对应ChargingPile的排队队列，从而建立QueueTicket与ChargingPile的关联。车辆完成批量调度后，需要从WaitingArea中移除，从而消除等候区关联。

问题4：哪个对象的属性需要修改或初始化？

QueueTicket初始化queue_Num和queue_Position。对应ChargingRequest的car_state从waiting修改为queued，并写入queue_Num与queue_Position。
