# B 部分问题暴露分析

**关联文档**：[B部分充电站桩管理审查报告.md](./B部分充电站桩管理审查报告.md)  
**模块**：`station/*`、`init_system`  
**日期**：2026-06-13

---

## 说明

本文档对 B 模块审查发现的 9 个问题，按**标准作业验收**与**日常运行**两类场景，说明：

- 哪些问题**不做临时改动也能通过验收**；
- 哪些问题在**平时运行**中可能被触发；
- 各问题的暴露条件与风险程度。

**范围提示**：B 模块负责系统配置与桩启停，**不参与**用户充电主流程（申请、排队、计费在 C/D 模块）。因此多数问题不会在「用户日常充电」中出现。

**标准验收依据**：[作业验收用例.md](./作业验收用例.md)（测至 09:30）、`test_acceptance_snapshot` / `run_acceptance`。验收事件中管理员操作 `(B,*,O,0/1)` 为**故障上报/恢复**，不含 `poweroff`、`poweron`、`start` 或修改 `system-config`。

---

## 一、标准验收：不做临时改动能否通过？

### 结论

**可以。** 在标准验收脚本路径下，B 模块 9 个问题**均不会单独导致验收失败**，无需为验收做临时 patch。

当前 `test_acceptance_snapshot` 已通过（含故障留桩、恢复重排、09:00 拒单等场景），从侧面印证上述判断。

### 逐项说明

| 问题 | 验收是否会踩中 | 原因 |
|------|----------------|------|
| **#1** `power_off` + 待重调度 | 否 | 验收**从不调用** `POST /api/pile/{id}/poweroff`，仅有 fault report/recover |
| **#2** 跳过 `standby` 直接 `start` | 否 | 验收**不执行**桩启停生命周期 |
| **#3** 改桩数不同步实体桩 | 否 | 验收**不修改** `system-config`，桩数固定 2 快充 + 3 慢充 |
| **#4** 非法 N/M 配置 | 否 | 验收**不修改**系统参数 |
| **#5** 心跳用墙钟时间 | 否 | 填表快照**不依赖** `last_heartbeat_at` |
| **#6** init 直接 `available` | 否（且有利） | 初始化后桩即可用，与用例中立刻有车充电一致 |
| **#7** 服务费双源 | 否 | 默认均为 0.8 元/度，与验收参数一致 |
| **#8** 测试断言过宽 | 否 | 仅影响自动化测试，不影响验收脚本 |
| **#9** 无事务包裹 | 否 | 验收事件**顺序、单线程**执行，无并发启桩场景 |

## 二、日常运行：哪些问题会暴露？

### 2.1 用户侧（注册、充电申请、排队、计费）

**9 项均不会在用户侧暴露。**

用户不调用 `/api/pile/*`、`/api/admin/system-config` 等 B 模块接口，上述问题与用户可见行为无关。

---

### 2.2 管理员 / 运维侧

| 问题 | 日常暴露程度 | 触发条件 | 可见现象 |
|------|--------------|----------|----------|
| **#1** | **中（特定操作）** | 故障桩上仍有 `pending_reschedule` 车辆时执行 **关桩** | `queue_used > 0` 仍可关桩成功；待重调度车辆挂在已关闭桩上 |
| **#2** | 低 | 对 `off` 桩直接调用 **投入运行**，跳过上电 | 功能可用，但与设计文档两步流程不一致 |
| **#3** | 低 | **`PUT` 修改** `fast_pile_num` / `slow_pile_num` | 配置数字与 `GET /api/pile/status` 桩列表不一致；`batch` 调度容量可能错位 |
| **#4** | 低 | **误填** `waiting_area_size=0`、负数等 | 配置更新成功，随后排队/调度整体异常 |
| **#5** | 极低 | **模拟时钟**开启且关注心跳时间 | 心跳时间与业务模拟时间不一致 |
| **#6** | 低（偏差异） | 演示「先启桩再充电」完整 UC | 初始化后桩已是 `available`，跳过启桩演示 |
| **#7** | 极低 | **手动改 DB** 或多处改价 | 配置页服务费与账单计算不一致 |
| **#8** | 无 | — | 仅影响 CI，不影响线上 |
| **#9** | 极低 | **高并发**同时启桩与调度 | 偶发桩状态与队列短暂不一致 |

---

### 2.3 按暴露场景分类

#### A. 标准验收路径 — 均不暴露（可不修也过验收）

#1、#2、#3、#4、#5、#6、#7、#8、#9

#### B. 用户日常充电 — 均不暴露

#1～#9（B 模块接口用户不可见）

#### C. 管理员日常运维 — 需关注

| 优先级 | 问题 | 说明 |
|--------|------|------|
| **需关注** | **#1** | 故障处理期间若误点关桩，会产生与界面/文档矛盾的行为 |
| 一般无感 | #2、#6 | 多路径或初始化策略差异，通常不报错 |
| 仅改配置时 | #3、#4 | 正常使用默认配置不会触发 |
| 边缘 | #5、#7、#9 | 模拟时钟、手改价、高并发 |

#### D. 刻意或误操作 — 容易暴露

- **#3、#4**：管理员修改系统参数
- **#2**：严格按 UC 演示启桩流程时与实现差异
- **#1**：故障未处理完毕就关桩

## 三、汇总对照表

| 问题 | 严重度 | 过标准验收 | 用户日常 | 管理员日常 | 建议 |
|------|--------|------------|----------|------------|------|
| #1 `power_off` | 高 | ✅ 不挡 | ✅ 不暴露 | ⚠️ 特定操作暴露 | **建议修复** |
| #2 跳过 standby | 中 | ✅ 不挡 | ✅ 不暴露 | 低 | 可文档化或收紧 |
| #3 桩数不同步 | 中 | ✅ 不挡 | ✅ 不暴露 | 低（改配置时） | 改配置场景再修 |
| #4 无数值校验 | 中 | ✅ 不挡 | ✅ 不暴露 | 低（误填时） | 改配置场景再修 |
| #5 时间源 | 中 | ✅ 不挡 | ✅ 不暴露 | 极低 | 与 A 联审后修 |
| #6 init 直接 available | 低 | ✅ 不挡 | ✅ 不暴露 | 低 | 可选 |
| #7 服务费双源 | 低 | ✅ 不挡 | ✅ 不暴露 | 极低 | 与 D 统一后修 |
| #8 测试断言 | 低 | ✅ 不挡 | ✅ 不暴露 | 无 | 测试改进 |
| #9 无事务 | 低 | ✅ 不挡 | ✅ 不暴露 | 极低 | 可选 |

图例：✅ 不挡/不暴露　⚠️ 特定条件下暴露

## 四、结论（可提交组长）

| 维度 | 结论 |
|------|------|
| **标准验收（至 09:30）** | B 模块 9 个问题**均可不修复而通过**；验收不测关桩、启桩链路与改配置 |
| **用户日常** | **均不暴露** |
| **管理员日常** | 主要风险为 **#1**（故障桩误关）；其余多为改配置、演示 UC 或极端场景 |
| **修复优先级** | **#1 仍建议修复**（违反 API 语义、有脏数据风险）；其余**不阻塞验收**，可按优先级排期 |
