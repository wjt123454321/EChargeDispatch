**结论**：存在 1 个高优先级缺陷 + 若干中低优先级设计与校验问题；分工建议的三项 API 测试均已通过。

| 严重度 | 数量 | 代表问题 |
|--------|------|----------|
| 高 | 1 | `power_off` 未计入 `pending_reschedule`，故障桩仍可关桩 |

**验收影响**：标准验收（至 09:30）**不调用**关桩/启桩/改配置，9 项问题均不单独导致验收失败。  
**日常风险**：管理员在故障桩仍有待重调度车辆时误点关桩（#1）为主要暴露点。

详见：

### 2.4 D — 充电请求、会话与计费（郭峰）

**结论**：存在待确认问题；必查逻辑基本符合，状态机、计费、等候区满返回等与文档一致。

| 序号 | 文件:行号 | 现象 | 严重度 | 建议 |
|------|-----------|------|--------|------|
| D-1 | `charging/services.py:343` | 等候区满时，事务内可能短暂超出容量后立刻派到有空位桩 | 低 | 需确认是否接受 |
| D-2 | `charging/services.py:343` | 并发下两请求同时读到 `waiting=9`，可能短暂超出等候区容量 | 中 | 是否加锁 |
| D-3 | `charging/services.py:553` | 充电中取消时 API 返回 `status` 为 `completed` | 低 | 确认是否改返回值 |
| D-4 | `frontend/.../request.js`、`useAuth.js` | 同浏览器两标签页分别登录用户/管理员，用户页轮询报权限不足 | 高 | localStorage 共享，需前后端协同 |
| D-5 | `charging/views.py:22`、`HomeView.vue:79` | 等候区满时 `accepted: false` 但前端仍提示「充电请求已提交」 | 中 | 前端需判断 `accepted` |
| D-6 | `billing/services.py:26` | 服务费计算精度 4 位，入库 2 位，小电量时详单与手算有偏差 | 中 | 确认是否统一精度 |

**需联审**：D-1/D-2 与 C 模块确认 `submit_request` 满员语义；D-4/D-5 涉及前端。
