# 智能充电桩调度计费系统联合验收对接开发标准

本文档用于联合验收时，指导第三方系统按照统一接口标准接入本系统。对方系统应优先使用 `外部对接接口` 完成站点状态查询、充电请求提交和请求状态查询；如验收需要覆盖完整用户流程，可补充调用 `用户端接口` 和 `管理员端接口`。

## 1. 基本信息

### 1.1 服务地址

本地调试地址：

```text
http://127.0.0.1:8080
```

公网验收地址：

```text
https://comgender-blog.top
```

接口统一前缀：

```text
/api
```

外部系统专用接口前缀：

```text
/api/v1/external
```

### 1.2 运行前置条件

联合验收前，系统提供方必须确认：

- 后端服务已启动，并监听 `8080` 端口。
- 公网域名 `https://comgender-blog.top` 已通过 Cloudflare Tunnel 转发到本机 `http://127.0.0.1:8080`。
- 数据库已启动，且系统已完成初始化。
- 已配置合作方认证环境变量，并重启后端服务使配置生效。

合作方认证环境变量：

```text
PARTNER_ID=<对方系统编号>
PARTNER_REMOTE_API_KEY=<对方调用本系统时使用的 API Key>
```

Windows PowerShell 设置示例：

```powershell
[Environment]::SetEnvironmentVariable("PARTNER_ID", "partner-a", "User")
[Environment]::SetEnvironmentVariable("PARTNER_REMOTE_API_KEY", "secret-a", "User")
```

设置后需要重新打开 PowerShell 或重启后端服务，使环境变量生效。

如果未配置上述变量，外部认证接口会返回：

```json
{
  "code": 1002,
  "message": "对外接口未配置",
  "data": null
}
```

### 1.3 数据格式

请求体统一使用 JSON：

```http
Content-Type: application/json
```

响应体统一格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

`code=0` 表示成功；非 0 表示失败。

### 1.4 通用错误码

| code | 含义 |
| --- | --- |
| 0 | 成功 |
| 1001 | 参数校验失败 |
| 1002 | 未登录、token 过期或外部认证失败 |
| 1003 | 权限不足 |
| 2001 | 用户不存在 |
| 2002 | 密码错误 |
| 2003 | 用户已注册 |
| 3001 | 已有未完成请求 |
| 3002 | 请求不存在 |
| 3003 | 当前状态不允许操作 |
| 3005 | 等候区已满 |
| 4001 | 充电桩不存在 |
| 4002 | 充电桩状态不允许操作 |
| 4003 | 充电桩无空位 |
| 6001 | 账单不存在 |

## 2. 统一业务枚举

### 2.1 充电模式

| 值 | 含义 |
| --- | --- |
| F | 快充 |
| T | 慢充 |

### 2.2 充电请求状态

| 值 | 含义 |
| --- | --- |
| queuing | 等候区排队中 |
| dispatched | 已进入充电区队列，等待开始充电 |
| charging | 正在充电 |
| pending_reschedule | 故障后等待重新调度 |

### 2.3 充电桩状态

| 值 | 含义 |
| --- | --- |

| available | 可用 |
| charging | 充电中 |
| fault | 故障 |

## 3. 外部系统认证标准

外部系统必须先用合作方凭据换取 `partner token`，再调用其他外部接口。

对方系统不得使用普通用户 token 或管理员 token 调用 `/api/v1/external/*` 接口。

`partner_id` 和 `api_key` 由本系统提供方在联合验收前分配。文档示例中的 `partner-a`、`secret-a` 仅为占位值，实际联调时必须替换为双方约定的真实值。

Token 有效期为 2 小时。过期后对方系统需要重新调用认证接口获取新 token。

### 3.1 获取外部接口 Token

```http
POST /api/v1/external/auth/token
```

请求：

```json
{
  "partner_id": "partner-a",
  "api_key": "secret-a"
}
```

成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "access_token": "<jwt-token>",
    "expires_in": 7200
  }
}
```

后续请求头：

```http
Authorization: Bearer <jwt-token>
```

## 4. 外部对接接口

### 4.1 查询充电站状态

用于对方系统获取当前充电桩状态、队列占用和等候区情况。

```http
GET /api/v1/external/piles/status
Authorization: Bearer <partner-token>
```

响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "station_id": "BUPT-01",
    "piles": [
      {
        "pile_id": 1,
        "mode": "F",
        "mode_label": "快充",
        "power": 30.0,
        "status": "available",
        "queue_used": 0,
        "queue_total": 5,
        "current_car": null,
        "total_charge_num": 0,
        "total_charge_time": 0.0,
        "total_charge_capacity": 0.0
      }
    ],
    "waiting_area": {
      "capacity": 10,
      "fast_waiting": 0,
      "slow_waiting": 0
    }
  }
}
```

### 4.2 提交充电请求

用于对方系统代车辆向本系统提交充电请求。本系统会按模式生成排队号，并在有空位时自动调度。

```http
POST /api/v1/external/charging/request
Authorization: Bearer <partner-token>
```

请求：

```json
{
  "car_id": "CAR-001",
  "request_mode": "F",
  "request_amount": 20.0,
  "callback_url": ""
}
```

字段说明：

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| car_id | 是 | string | 车辆/用户唯一编号，最长 20 字符 |
| request_mode | 是 | string | `F` 快充，`T` 慢充 |
| request_amount | 是 | number | 请求充电量，必须大于 0 |
| callback_url | 否 | string | 预留字段，当前版本不触发回调 |

成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": 12,
    "queue_num": "F3",
    "status": "dispatched",
    "pile_id": 1,
    "position": {
      "position": "充电区",
      "queue_num": "F3"
    }
  }
}
```

注意：

- 对方提交的 `car_id` 如不存在，本系统会自动创建一个默认外部用户。
- 同一车辆存在未完成请求时，会返回 `3001`。
- 等候区满或批量调度等待集合已满时，会返回 `3005`。
- `request_id` 是后续查询请求状态的唯一凭据，对方系统必须保存。
- 当前版本不支持异步回调，对方系统应通过 `4.3 查询请求状态` 轮询。

### 4.3 查询充电请求状态

```http
GET /api/v1/external/charging/status/{request_id}
Authorization: Bearer <partner-token>
```

该接口严格按 `request_id` 查询指定请求。即使同一车辆后续提交了新的充电请求，也不会影响旧 `request_id` 的查询结果。若请求已完成或已取消，`position` 可能返回 `null`。

响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "request_id": 12,
    "queue_num": "F3",
    "status": "dispatched",
    "request_mode": "F",
    "request_amount": 20.0,
    "pile_id": 1,
    "position": {
      "position": "充电区",
      "queue_num": "F3"
    },
    "fault_notice": false
  }
}
```

### 4.4 外部接口能力边界

当前 `/api/v1/external` 接口用于联合验收的最小系统对接，覆盖：

- 获取合作方 token。
- 查询充电站和充电桩状态。
- 代车辆提交充电请求。
- 根据 `request_id` 查询请求状态。

当前 `/api/v1/external` 不直接提供以下能力：

- 开始充电。
- 结束充电。
- 取消充电。
- 查询账单和详单。
- 触发故障上报或故障恢复。

如联合验收需要覆盖完整用户业务闭环，对方系统应调用第 5 节的用户端接口；如需要覆盖管理端故障和报表流程，应调用第 6 节的管理员接口。

## 5. 用户端接口标准

当对方系统需要模拟完整用户客户端时，可调用以下接口。

### 5.1 用户注册

```http
POST /api/auth/register
```

```json
{
  "car_id": "CAR-001",
  "user_name": "张三",
  "car_capacity": 60,
  "password": "123456"
}
```

### 5.2 用户登录

```http
POST /api/auth/login
```

```json
{
  "car_id": "CAR-001",
  "password": "123456"
}
```

成功后使用返回的 `access_token` 调用用户接口。

### 5.3 提交充电请求

```http
POST /api/charging/request
Authorization: Bearer <user-token>
```

```json
{
  "request_mode": "F",
  "request_amount": 20
}
```

### 5.4 修改请求充电量

仅允许车辆仍在等候区 `queuing` 时修改，排队号不变。

```http
PUT /api/charging/amount
Authorization: Bearer <user-token>
```

```json
{
  "amount": 25
}
```

### 5.5 修改充电模式

仅允许车辆仍在等候区 `queuing` 时修改。修改模式后会重新生成排队号，并排到新模式队列末尾。

```http
PUT /api/charging/mode
Authorization: Bearer <user-token>
```

```json
{
  "mode": "T"
}
```

### 5.6 查询排队状态

```http
GET /api/charging/queue-status
Authorization: Bearer <user-token>
```

### 5.7 开始充电

车辆必须已经被调度到充电区，并且必须是对应充电桩队列第一位。

```http
POST /api/charging/start
Authorization: Bearer <user-token>
```

```json
{
  "pile_id": 1
}
```

### 5.8 查询充电状态

```http
GET /api/charging/status
Authorization: Bearer <user-token>
```

### 5.9 结束充电

结束后系统生成详单和账单。

```http
POST /api/charging/end
Authorization: Bearer <user-token>
```

### 5.10 取消充电

等候区、充电区均允许取消；若正在充电，会按已充电量生成详单。

```http
DELETE /api/charging/cancel
Authorization: Bearer <user-token>
```

### 5.11 账单与详单

查询账单：

```http
GET /api/bill/list
Authorization: Bearer <user-token>
```

查询详单：

```http
GET /api/bill/detail/{bill_id}
Authorization: Bearer <user-token>
```

支付账单：

```http
POST /api/bill/pay/{bill_id}
Authorization: Bearer <user-token>
```

## 7. 调度与计费规则标准

### 7.1 普通调度

系统按充电模式分别调度：

- 快充请求 `F` 只能进入快充桩。
- 慢充请求 `T` 只能进入慢充桩。
- 当对应模式充电桩存在空位时，系统从该模式等候队列队首叫号。
- 被叫号车辆会进入“完成时间最短”的充电桩队列。
- 完成时间 = 该桩队列中所有车辆剩余充电时间之和 + 当前车辆充电时间。

### 7.2 故障调度

管理员可在系统配置中选择：

```text
priority   优先级调度
time_order 时间顺序调度
```

故障发生时：

- 正在充电车辆立即停止计费并生成详单。
- 故障桩队列中未完成车辆进入重新调度流程。
- 重新调度期间暂停等候区普通叫号。

### 7.3 扩展调度

管理员可在系统配置中选择：

```text
normal
single_min_total
batch_min_total
```

`single_min_total`：

- 同模式多个空位同时出现时，可以一次叫多个号。
- 不考虑这批车辆内部先后顺序。
- 目标是这批车辆完成充电总时长最短。

`batch_min_total`：

- 只有当到达车辆数量达到全站容量时才开始批量调度。
- 不区分快充和慢充偏好。
- 所有车辆可分配任意类型充电桩。
- 目标是一批车辆完成充电总时长最短。

### 7.4 计费规则

充电费：

```text
充电费 = 单位电价 * 实际充电度数
```

服务费：

```text
服务费 = 0.8 * 实际充电度数
```

总费用：

```text
总费用 = 充电费 + 服务费
```

分时电价：

| 时段 | 时间 | 单价 |
| --- | --- | --- |
| 峰时 | 10:00-15:00，18:00-21:00 | 1.0 元/度 |
| 平时 | 7:00-10:00，15:00-18:00，21:00-23:00 | 0.7 元/度 |
| 谷时 | 23:00-次日 7:00 | 0.4 元/度 |

跨时段充电时，系统会按时间边界拆分计费。

## 8. 联合验收推荐流程

### 8.1 最小外部系统对接流程

1. 对方调用 `/api/v1/external/auth/token` 获取 partner token。
2. 对方调用 `/api/v1/external/piles/status` 查询站点状态。
3. 对方调用 `/api/v1/external/charging/request` 提交车辆充电请求。
4. 对方记录返回的 `request_id` 和 `queue_num`。
5. 对方调用 `/api/v1/external/charging/status/{request_id}` 轮询请求状态。

### 8.2 完整用户流程联调

1. 注册或登录用户。
2. 提交充电请求。
3. 查询排队状态。
4. 若状态为 `dispatched`，调用开始充电。
5. 充电一段时间后调用结束充电。
6. 查询账单和详单。

### 8.3 管理员故障流程联调

1. 管理员登录。
2. 查询充电桩状态。
3. 配置故障调度策略。
4. 上报某充电桩故障。
5. 查询故障记录和车辆重调度结果。
6. 恢复故障桩。
7. 查询恢复后的队列状态。

## 9. 对方系统开发约束

对方系统必须遵守：

- 所有请求均使用 JSON。
- 所有受保护接口必须带 `Authorization: Bearer <token>`。
- 外部对接接口只能使用 partner token。
- 不要依赖 `callback_url`，当前版本没有回调通知。
- 不要假设请求提交后一定立即进入充电区，必须根据返回状态和查询接口判断。
- 不要自行计算队列号，应以本系统返回的 `queue_num` 为准。
- 不要自行计算费用，最终费用以本系统账单/详单为准。
- 时间字段按北京时间返回，格式通常为 ISO 字符串。
